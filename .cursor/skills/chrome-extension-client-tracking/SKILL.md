# Chrome Extension Client Tracking

## Когда применять

Применяй этот навык, если запрос включает одно или несколько:
- Chrome Extension, `manifest.json`, Manifest V3, content script, options page
- Наблюдение за DOM (`MutationObserver`) и парсинг никнейма/профиля на странице
- Внедрение UI в существующее модальное окно сайта
- Supabase или Postgres, REST API и авторизацию по `Secret Key`
- Хранение комментариев, количества сделок, проверку уникальности клиента

## Базовый стандарт реализации

1. Используй Manifest V3:
   - `manifest_version: 3`
   - минимальные permissions (например, `storage`)
   - `host_permissions` только для нужных доменов
   - `content_scripts` с точными `matches` (без лишних wildcard)
2. Делай проверку домена в рантайме в content script:
   - если `location.hostname` не из allowlist, заверши выполнение
3. Храни `Secret Key` только в `chrome.storage.local`.
4. Получай ключ через options page (форма + валидация пустого значения).
5. Работай с DOM безопасно:
   - используй `MutationObserver` с debounce/throttle
   - минимально вмешивайся в DOM сайта
   - избегай дублирования внедренных элементов (ставь `data-*` маркеры)
6. Для API:
   - используй `GET/POST/PUT/PATCH` по назначению
   - передавай `Secret Key` в заголовке авторизации
   - добавляй таймаут, `try/catch`, понятные ошибки для UI
7. После сохранения данных обновляй UI сразу (optimistic update либо мгновенный refetch).

## DOM и UI правила

- Внедряй элементы только в стабильный контейнер (например, корень модального окна).
- Никогда не меняй существующие обработчики сайта без крайней необходимости.
- Для повторных рендеров:
  - проверяй, внедрен ли уже компонент (`querySelector` по `data-ext-id`)
  - переиспользуй существующий узел, а не создавай новый
- Состояние кнопки/иконки должно быть очевидным:
  - `loading`, `saved`, `error`, `stale`
- Анимации должны быть короткими и ненавязчивыми (`150-250ms`, ease/ease-out).

## API и данные

- Поддерживай единый формат клиента:
  - `nickname` (string, normalized)
  - `comment` (string)
  - `dealsCount` (number)
  - `updatedAt` (ISO datetime)
- Проверка уникальности:
  - нормализуй никнейм (trim + lowercase)
  - перед upsert проверь существующую запись
- Операции:
  - `GET` поиск/чтение по никнейму
  - `POST` создание
  - `PUT/PATCH` обновление комментария и числа сделок
- Логирование:
  - в `console` только технический минимум
  - не логируй `Secret Key` и чувствительные payload-поля

## Security / Best Practices

- Не хардкодь ключи, токены и URL с секретами.
- Не выводи секреты в UI, логи, ошибки, screenshot-friendly места.
- Держи принцип наименьших привилегий в `permissions` и `host_permissions`.
- Для ошибок показывай пользователю безопасное сообщение:
  - "Не удалось сохранить данные. Повторите попытку."
  - детали ошибки оставляй только в debug-логах.

## Что делать при неоднозначности

- Если неясен источник никнейма в DOM, сначала найди 2-3 стабильных селектора и выбери самый устойчивый.
- Если API-контракт не дан, предложи минимальный контракт `GET + POST + PATCH` и согласуй поля.
- Если Supabase/Postgres не выбран, по умолчанию проектируй нейтральный REST-слой без vendor lock-in.
